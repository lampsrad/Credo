📈 Project Overview

CredoApp is a personal investment portfolio tracker built on ASP.NET Core Blazor (Interactive Server) targeting .NET 10. It persists securities, transactions, and daily price history in a local SQL Server database and synchronises live market data from the Yahoo Finance API via the YahooQuotesApi library. Its purpose is to track holdings, compute annualised returns (XIRR), and visualise portfolio value over time against the S&P 500 benchmark.

Framework

ASP.NET Core Blazor — Interactive Server mode (SignalR-based live UI).

Language / Runtime

C# 13 on .NET 10.0 — async/await throughout, DateOnly everywhere.

Database

Microsoft SQL Server via Entity Framework Core 10 (code-first, no migrations).

Market Data

YahooQuotesApi v7 — snapshots, historical daily series, P/E, dividend yield, FX rates.

Charting

Chart.js (CDN) driven by JS interop. Toggles for 50/200-DMA and price-vs-% view.

CSV Import

CsvHelper v33 with custom ClassMaps and null-tolerant type converters.

🗃 Database & EF Core

CredoDbContext declares one DbSet<T> per entity. There is no OnModelCreating; relationships are inferred by convention from the SecurityID, TickerID, CurrencyID foreign-key properties.

DbContextFactory

DbContextFactory.cs resolves the connection string in this order, returning the first non-empty match:

  1. appConfig.ConnectionKey (default "XPS", mutable at runtime)
  2. appConfig.MachineName (the value of Environment.MachineName)
  3. literal "Local"

Within the chosen template it substitutes placeholders:

// Template in appsettings.json
"Server=[ServerName];Database=[DatabaseName];Trusted_Connection=True;
 TrustServerCertificate=True;MultipleActiveResultSets=True"

// At runtime:
//   [ServerName]   → {MachineName}\SQL
//   [DatabaseName] → appConfig.DbName  (defaults to "Credo")

In Development mode the builder enables EnableSensitiveDataLogging().

📊 History Table — How It's Compiled

The History table is the time-series backbone of the application. Every row is a (Symbol, Date, Price) triple. The table holds two logically distinct kinds of series stored side by side:

  • Per-ticker price series — Symbol is a Yahoo symbol (e.g. AAPL, VOD.L, ^GSPC, GBPUSD=X). Used for individual ticker charts, the SPY benchmark, FX-rate lookup during portfolio recomputation, and as input to the synthetic portfolio series.
  • Portfolio total series — Symbol = "Portfolio" (capital P). One row per day equal to the total USD market value of all holdings on that day.

The (Symbol, Date) pair

There is no unique index on (Symbol, Date) in the database — duplicate-prevention is enforced entirely in C# by building a HashSet<(string, DateOnly)> of existing keys at the start of an update and using HashSet.Add as an atomic "does-not-already-exist" test.

Three entry points that write to History

1Daily append
UpdateService.UpdateSecuritiesAsync  → AppendPortfolioHistoryAsync & AppendSecuritiesHistoryAsync. Upserts today's row for each security plus the portfolio total.
2Bulk backfill
UpdateService.UpdateHistoryAsync — for each ticker, fetches Yahoo daily history from the last known date forward, appends rows not already present.
3Portfolio rebuild
GraphService.BackfillPortfolioHistoryAsync (not wired to UI). Wipes the entire Symbol="Portfolio" series and rebuilds it from transactions × per-ticker history × FX.

One read entry point also matters for understanding History: GraphService.ComputePortfolioChartAsync performs the same synthetic-portfolio computation as path 3 but in memory only — used when the user excludes one or more securities from the portfolio chart on the PortfolioView page.

📅 1 · Daily Append (the "Update All" path)

Triggered when the user clicks Update › Update All in the menu, or Refresh on the PortfolioView. The single public entry point is UpdateService.UpdateSecuritiesAsync(); everything else in the chain is private.

End-to-end pipeline

1Read tickers
All Ticker rows except USDUSD=X (the self-USD pseudo-FX is excluded everywhere).
2Yahoo snapshot
YahooQuotesBuilder().Build().GetSnapshotAsync(symbols). A 1-second PeriodicTimer bumps the progress bar in 10-point steps up to 95 while waiting.
3Update Securities
For each security, copy Price, PrevClose, Exchange, ChangePercent, DividendYield, EPS, PE, PEf, MarketCap, Currency from the snapshot. Missing snapshot → Price = 1.
4Update Currencies
For each Currency (except USDUSD=X), copy the snapshot RegularMarketPrice into Rate.
5SaveChanges
Commits the Security + Currency updates as one transaction.
6UpdatePortfolio
Recomputes Portfolio rows from the just-updated prices & FX rates (Market_Value, Gain, GainPerc, Pct, IRR).
7AppendPortfolioHistory
Upserts today's Symbol="Portfolio" row to the new total.
8AppendSecuritiesHistory
Upserts today's row for every security with a known price.

AppendPortfolioHistoryAsync — the portfolio row

private async Task AppendPortfolioHistoryAsync()
{
    await using var scope = repo.BeginScope();
    var ports = await scope.GetEntitiesAsync<Portfolio>();
    var total = Math.Round(ports.Sum(p => p.Market_Value ?? 0m), 2);
    var today = DateOnly.FromDateTime(DateTime.Today);

    var row = await scope.GetEntityAsync<History>(
        h => h.Symbol == "Portfolio" && h.Date == today);

    if (row is null)
        scope.Add(new History { Symbol = "Portfolio", Date = today, Price = total });
    else
        row.Price = total;                    // ← upsert: overwrite same-day value

    await scope.SaveChangesAsync();
}

Two non-obvious properties of this routine:

  • Upsert semantics. Running Update All multiple times in one day does not create duplicate rows — it just updates today's existing one. This is why the "natural key" is (Symbol, Date) even without a DB constraint.
  • Depends on UpdatePortfolio having just run. The total is read from Portfolio.Market_Value, not recomputed from securities directly. Calling this method standalone (without the preceding portfolio recompute) would record yesterday's total against today's date.

AppendSecuritiesHistoryAsync — per-security rows

foreach (var security in securities)
{
    var symbol = security.ticker?.Symbol;
    if (symbol is null) continue;

    var row = await scope.GetEntityAsync<History>(
        h => h.Symbol == symbol && h.Date == today);

    if (row is null)
        scope.Add(new History { Symbol = symbol, Date = today, Price = security.Price });
    else
        row.Price = security.Price;
}
await scope.SaveChangesAsync();

Same upsert pattern. Note that securities filter is x.ticker != null && x.Price != null — securities without a resolved ticker (e.g. cash positions like "AUSTRALIAN DOLLAR") never produce a History row. The Price here is the value just written by UpdateSecuritiesAsync, i.e. the latest Yahoo snapshot price in the security's local currency.

Why two separate methods instead of one? The portfolio total is denominated in USD (already FX-converted in Portfolio.Market_Value), while per-security prices are in each security's own currency. They are different units stored in the same table, distinguished only by the Symbol column.

📥 2 · Bulk Backfill from Yahoo (UpdateHistoryAsync)

Triggered from Update › Securities History. Fills gaps in the per-ticker daily price series — not the "Portfolio" series. Idempotent.

Algorithm

  1. Read all Ticker rows. If there are none, bail out.
  2. Read the entire History table once. Build two structures from it:
    • existingKeys — HashSet<(string, DateOnly)> for O(1) duplicate detection.
    • lastDateBySymbol — Dictionary<string, DateOnly> mapping each symbol to its latest known date.
  3. For each ticker, pick a start date: 
    start = lastDateBySymbol.TryGetValue(symbol) ? lastKnownDate : today minus 5 years
    The default of 1 August 2016 is the project's epoch — the earliest date the user wants history for if a symbol is brand new.
  4. Build a fresh YahooQuotesBuilder per ticker with the start date converted into a NodaTime Instant and fetch GetHistoryAsync(symbol).
  5. For each returned tick (sorted chronologically), call existingKeys.Add((symbol, date)). If it returns true, the key was not previously present → stage a new History row. If false, skip. This deduplicates against both the database and rows already staged in the current run.
  6. Tickers whose symbols Yahoo rejects throw ArgumentException and are silently skipped (see FetchTickerHistoryAsync).
  7. After looping all tickers, perform a second Yahoo call: GetSnapshotAsync(allSymbols). For each symbol that does not already have a row for today, add one using snapshot.RegularMarketPrice.
  8. One bulk AddRange(toAdd) + SaveChangesAsync().
Why the second snapshot call at the end? Yahoo's daily history endpoint does not publish "today" until after market close — and for indices like ^GSPC, often not even then. Without the snapshot top-up, charts would always be missing the most recent point, and the portfolio backfill would also stop one day short. The snapshot gives every ticker the same end-of-day treatment that AppendSecuritiesHistoryAsync gives the user's own securities.

Progress reporting

state.UpdateProgress(done * 100.0 / total, $"Updating History ({done}/{total})");

State.UpdateProgress takes a numeric percent (clamped 0–100) and an optional title — not a free-form message. The progress UI reads State.ProgressVal and State.TitleD from the event payload.

📈 3 · Portfolio Series Rebuild (Backfill & in-memory)

The synthetic "Portfolio" series can be rebuilt from scratch by reconstructing what the portfolio's USD market value would have been on every historical date. Two methods share the same algorithm — one writes to History, the other returns the result as ChartData for live "exclude-some-securities" views.

MethodOutputWhen invoked
GraphService.BackfillPortfolioHistoryAsync() Wipes all Symbol="Portfolio" rows and reinserts the entire series. Returns row count. Not wired to UI — manual one-off (e.g. after a corrupted History). The method header explicitly notes "Future Use".
GraphService.ComputePortfolioChartAsync(excludedSecurityIds?) Returns in-memory ChartData; never touches the DB. Called from PortfolioView.ShowPortfolioChartAsync whenever the user has ticked any rows to exclude. With no exclusions, the page reads the pre-computed series via LoadTickerDataAsync("Portfolio", null).

Shared algorithm

  1. Read all Security rows. Find the earliest buy-transaction date across the lot (using IsBuy: by or li, case-insensitive). This is the first date on which the portfolio existed.
  2. Build the list of relevant symbols:
    • secSymbols — every distinct ticker.Symbol present.
    • fxSymbols — every distinct security Currency that is not USD=X.
    Fetch all History rows whose Symbol matches either set, ordered by Date.
  3. Group History by symbol into a Dictionary<string, List<History>> for fast lookup.
  4. Enumerate the union of distinct Dates in that history (between earliest buy and today). This is the date axis of the resulting series — note it is only dates on which some ticker traded, not a continuous calendar.
  5. For each date, sum across securities:
    1. Replay this security's transactions up to and including the date to derive the open quantity (+by/+li, -sl/-lo). Skip if ≤ 0.
    2. Look up the security's price via LastOnOrBefore(history, date) — the most recent price row at or before the date (handles weekends, holidays, and tickers that haven't traded yet).
    3. If the security's currency ≠ USD=X, do the same LastOnOrBefore lookup against the FX symbol to get the conversion rate. Skip the security on dates where no FX rate is yet known.
    4. Add qty × price × fx to the day's total.
  6. Round each daily total to 2 dp and emit a label/value pair (or a History row).

LastOnOrBefore — the "last known price" rule

private static History? LastOnOrBefore(List<History> sortedRows, DateOnly date)
{
    History? best = null;
    foreach (var r in sortedRows)
    {
        if (r.Date > date) break;
        best = r;
    }
    return best;
}

Linear scan through pre-sorted rows. Because the input is grouped + sorted once per request, the total cost is acceptable for typical history sizes. This is what allows the portfolio total on a Saturday to still use Friday's close. If a security's symbol has no price on or before the date (it hadn't started trading yet), that security simply contributes zero that day.

Backfill is destructive. BackfillPortfolioHistoryAsync begins by deleting all rows where Symbol = "Portfolio" in a single RemoveRange call, then re-inserts the recomputed series. There is no transaction rollback safeguard beyond EF's per-call transaction — a backup before running it is wise.

💹 XIRR Calculator

Credo/XirrCalculator.cs. Newton-Raphson solver (max 200 iterations, tolerance 1e-8) with a bisection fallback bracketed at [-0.999, 100.0]. Years are computed via DateOnly.DayNumber / 365.0.

// by, li               → outflow (negative LocalAmount)
// sl, lo, dv, in, cv  → inflow  (positive LocalAmount)
// Unrecognised codes  → silently skipped
MethodDescription
CalculateXirr(IReadOnlyList<(DateOnly, decimal)>)Solver on raw cash flows.
CalculateXirr(IList<Transaction>)Maps TranCodes to signed cash flows, then solves. Caller appends a synthetic cv close-out at Quantity × current price for open positions.
CalculateSpyXirr(transactions, spyPrices, currencyRate?)Hypothetical XIRR if the same by/li/sl/lo cash had bought SPY at the next available date. Adds a residual at the latest SPY date for unsold SPY shares.

📤 Import / Export

All imports are triggered from /import (ImportView.razor). Each button turns red after it is clicked so you can see at a glance what has already run in the current session. After every import a success banner shows the number of rows written to the database and disappears automatically after five seconds. All CSV parsing uses the CsvHelper library with class maps defined in Classes/Classes.cs.

Recommended import order

There are foreign-key dependencies between the tables. Always run the buttons in this sequence on a fresh database — skipping or reversing steps will leave orphaned records:

  1. Tickers — master list of ticker symbols; everything else references it.
  2. Currencies — currency lookup table referenced by Securities and Portfolio.
  3. Transactions — raw ledger rows; stored by security name (string) at this stage.
  4. Securities — creates Security rows from Tickers and back-links Transactions by name.
  5. Portfolio — current holdings snapshot; links to Securities and Currencies.

1 · Tickers — ImportTicker()

File: first file matching Ticker*.csv in AppConfig.DownloadsPath.
Class map: TickerMap

CSV columnMaps toNotes
tickerTicker.SymbolYahoo Finance ticker symbol, e.g. AAPL
securitynameTicker.NameHuman-readable company name
currencyTicker.CurrencyISO currency code, e.g. USD, EUR

Deduplication: all existing Symbol values are loaded into a HashSet<string> before the loop. Any CSV row whose symbol is already present is silently skipped — existing tickers are never updated by this import. Only genuinely new symbols are inserted. This makes the operation safe to re-run at any time.

2 · Currencies — ImportCurrencies()

File: first file matching Currencies*.csv in DownloadsPath.
Class map: CurrencyMap

CSV columnMaps toNotes
currencyCurrency.NameShort name, e.g. USD, GBP
symbolCurrency.SymbolYahoo FX symbol, e.g. USD=X, GBP=X

Upsert logic: for each CSV row the code checks whether a Currency with the same Symbol already exists in the database (tracked query inside a RepoScope).

  • New symbol — inserted as a new row. If the currency name is "USD" the Rate is forced to 1 (base currency); all other currencies get the rate that was in the CSV (typically 0 until the first Yahoo FX update).
  • Existing symbol — only Symbol and Name are refreshed. The live Rate that Yahoo FX updates write is intentionally left untouched.

3 · Transactions — ImportTransactions() / ImportTransactionsAsync()

File: first file matching Transaction*.csv in DownloadsPath.
No class map — the CSV is read field-by-field because the file contains preamble rows above the real header.

Header detection: the reader scans rows one by one until it finds a row where column index 1 equals "Tran Code". That row is treated as the header and ReadHeader() is called. All rows before it are ignored. This is robust against brokerage exports that prepend account-level summary lines.

CSV columnMaps toProcessing
Tran CodeTransaction.TranCodeRows with a blank code or a code longer than 4 characters are skipped (footer/summary lines).
Trade DateTransaction.TradeDateDateRegex() normalises D/M/YY and D/M/YYYY to YYYY-MM-DD before DateOnly.TryParse. Rows with unparseable dates are skipped.
QuantityTransaction.QuantitySpaces (thousands separators) and trailing periods are stripped. Decimal values are rounded to int.
Local CurrencyTransaction.CurrencyStored as-is, e.g. USD.
Local AmountTransaction.LocalAmountSpaces stripped; parsed with NumberStyles.Any and InvariantCulture.
DescriptionTransaction.DescriptionPlain text, stored verbatim.
SecurityTransaction.SecuritySecurity name string — the FK to SecurityID is not set here; that link is created by ImportSecurities.

Deduplication: all existing (TradeDate, LocalAmount) pairs are loaded into a HashSet before the loop. A new row is only inserted when that exact pair is not already present. This prevents double-imports when the same transaction file is processed twice, while still allowing two legitimate transactions on the same date if their amounts differ.

4 · Securities — ImportSecurities()

No file required — this import does not read a CSV. It derives Security rows entirely from the Ticker table that was populated in step 1, then wires up the transactions imported in step 3.

Phase 1 — Create Securities from Tickers:

  • Loads all existing Ticker rows (tracked).
  • Loads all Currency rows and builds a Dictionary<Name, ID> lookup.
  • For each ticker, creates a Security with:
    • TickerID → the ticker's primary key.
    • SecurityName → Ticker.Name.
    • CurrencyID → looked up from the ticker's Currency string via the currency dictionary (null if not found).
  • All new Security rows are bulk-inserted with AddRange.

Phase 2 — Back-link Transactions:

  • Loads all Transaction rows whose Security name matches any of the newly created security names.
  • Groups transactions by their Security name string.
  • For each Security, sets Transaction.SecurityID to the new Security's Id on every matching transaction.
  • A second SaveChangesAsync persists the FK updates.

⚠ Run once per ticker batch. If you later import more Tickers and run ImportSecurities again, it will attempt to insert duplicate Securities because there is no symbol-level deduplication guard in this step. Export Securities to CSV first and check for conflicts before re-running.

5 · Portfolio — ImportPortfolio()

File: first file matching Portfolio*.csv in DownloadsPath.
Class map: PortfolioMap

CSV columnMaps toNotes
QuantityPortfolio.QuantityDecimal in CSV, rounded to int.
Security DescriptionPortfolio.Security_DescriptionUsed as the match key for upsert and Security lookup.
Unit CostPortfolio.Unit_CostCost per share at time of purchase.
PricePortfolio.PriceCurrent market price per share.
CostPortfolio.CostTotal cost basis (Quantity × Unit Cost).
Market ValuePortfolio.Market_ValueCurrent market value (Quantity × Price).
PctPortfolio.Pct% sign stripped before decimal parse.
P&L %Portfolio.GainPerc% sign stripped before decimal parse.

Row filtering: rows are silently skipped when:

  • Market_Value == 0 — zero-value lines (cash sweeps, expired options, etc.).
  • Security_Description == "CONSTELLATION SOFTWARE-WT 40" — a hardcoded exclusion for a known warrant entry that has no tradeable equivalent.

Upsert logic:

  • Existing row (matched by Security_Description): if Unit_Cost == Price (meaning the position was just opened and cost equals market price), the Quantity is updated. Otherwise the row is left unchanged — the live price & market value from Yahoo updates take precedence.
  • New row: a Security is looked up by SecurityName == Security_Description. If found, the new Portfolio row inherits SecurityID, CurrencyID, and Price from the Security. If no Security match exists (e.g. a cash or FX line), MapCurrencyName() derives a currency from the first word of the description: Australian → USD, Euro → EUR, Pound → GBP, US → USD, anything else → USD. Price is set to 1.

Shared infrastructure

CsvConfiguration (CsvConfig())

All five import paths reuse the same CsvConfiguration:

  • Header record expected (HasHeaderRecord = true).
  • Comma delimiter, InvariantCulture.
  • TrimOptions.Trim — leading/trailing whitespace stripped from every field.
  • Blank lines ignored; bad data and missing fields do not throw — they return null.

Custom type converters

ConverterBehaviour
DecimalNullConverterStrips spaces (thousands separator), then parses with NumberStyles.Any + InvariantCulture. Empty or unparseable input → null.
IntegerNullConverterSame stripping logic, parses to int. Empty or unparseable → null.
CustomDateConverterTries yyyy-MM-dd exact match first; falls back to DateTime.TryParse with InvariantCulture. Returns a DateOnly, or null on failure.

DateRegex() — date normalisation

Used exclusively by the Transaction import. Matches the pattern \b(\d{1,2})/(\d{1,2})/(\d{2,4})\b and rewrites every match as YYYY-MM-DD, zero-padding day and month and expanding two-digit years with a "20" prefix. The function is called twice on the same string to handle edge cases where two date-like tokens appear in the same field.

Feedback — ShowMessage()

After every import ShowMessage(int cc) is called with the EF Core SaveChangesAsync return value (number of rows affected). It sets results.Records, triggers a re-render, waits 5 seconds via Task.Delay(5000), then resets results and triggers another re-render — giving a self-dismissing confirmation banner without any JavaScript.

Export

Menu.razor.cs exports Securities and Tickers as plain CSV files written with StreamWriter (no CsvHelper on the write path) directly into AppConfig.DownloadsPath. These export files can be edited and re-imported via the Tickers button if symbols need to be corrected.

🔎 Watchlist

The Watchlist lets you track any Yahoo Finance symbol — price history, chart, S&P 500 overlay — without creating a Security, Ticker, or Portfolio entry. Use it to monitor candidates before buying, follow sold positions, or benchmark against any index.

/watchlist  ·  WatchlistView.razor

Data model

A single Watchlist table (WatchlistItems in SQL Server) holds the registry of symbols to track:

ColumnTypePurpose
IdINT IDENTITY PKSurrogate key
SymbolNVARCHAR(50)Yahoo Finance ticker, e.g. META
NameNVARCHAR(300)Human-readable name; auto-fetched from Yahoo if left blank on add

Price history is stored in the existing History table — the same table used by Securities and the Portfolio series. No separate history table is needed; the Symbol column is the only key.

The table is created automatically on startup via an idempotent IF NOT EXISTS … CREATE TABLE statement in Program.cs — no EF Core migration is required.

Adding a symbol

  1. Type the Yahoo Finance ticker into the Symbol box (e.g. META).
  2. Optionally type a display name. If left blank, Credo fetches LongName from a Yahoo snapshot call automatically.
  3. Press Enter or click ＋ Add.
  4. The symbol is checked for duplicates; a warning is shown if it is already on the list.

Fetching history — UpdateWatchlistHistoryAsync()

Click ⟳ Update History on the Watchlist page, or use Update → Watchlist History from the menu. The method:

  1. Loads all Watchlist rows.
  2. Builds a HashSet<(Symbol, Date)> from existing History rows to skip already-stored data.
  3. For each symbol, determines the last stored date and fetches only the incremental range from Yahoo (back-fill to 5 years before today on first run).
  4. Appends a today-snapshot via GetSnapshotAsync to cover the current trading day before Yahoo publishes the official close.
  5. Bulk-inserts new rows and saves.

This reuses the same FetchTickerHistoryAsync helper used by UpdateHistoryAsync for Securities, so incremental updates are efficient — only new dates are fetched on subsequent runs.

Charting

The chart button (📈) on each row calls GraphService.LoadAdhocTickerDataAsync, which has a fast path: if History rows already exist for the symbol the chart renders instantly from the database. The live Yahoo network call is only made for symbols that have never been fetched.

The full ChartModal is shown — including the S&P 500 overlay, 50/200 DMA toggles, and percentage/price toggle — exactly the same as for portfolio securities.

Removing a symbol

Click ✕ on any row. This removes the Watchlist registry entry only. History rows are intentionally left in place so that re-adding the symbol later resumes from where it left off rather than re-downloading years of data.

☀ Key Design Patterns

No-Track Reads

Repo reads all use AsNoTracking for display-only queries — avoids EF change-tracking overhead and stale cached entities.

Performance

Unit-of-Work Writes

RepoScope owns one tracked DbContext per write workflow and disposes it as soon as SaveChangesAsync completes.

Data Integrity

(Symbol, Date) Natural Key

History uniqueness is enforced in code via HashSet<(string, DateOnly)>. Daily appends are upserts; bulk fetches skip existing.

History

LastOnOrBefore Pricing

Portfolio rebuild handles weekends/holidays/IPO gaps by always pulling the most recent price & FX at or before each date.

History

Today-Snapshot Top-up

Yahoo daily history lags one day. After backfill, a separate GetSnapshotAsync call inserts today's row for every ticker.

History

Event-Driven UI State

State uses delegate events as a loose-coupled bus between services and UI.

Decoupling

Machine-Aware Config

Connection strings keyed by machine name (ROG / XPS / Local) so the same codebase runs everywhere without conditional code.
